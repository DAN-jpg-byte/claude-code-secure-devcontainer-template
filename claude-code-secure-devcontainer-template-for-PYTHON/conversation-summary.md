# 会話まとめ（テンプレ運用メモ）

## 1. このテンプレートの基本方針

- 目的は、Claude Code を DevContainer 内で安全に使うこと。
- セキュリティは主に以下の多層防御で構成される。
  - DevContainer による分離
  - `.env` をコンテナ外に置き、`--env-file` で環境変数注入
  - `.claude` 設定のマウントで設定・認証を永続化
  - `init-firewall.sh` による外向き通信制限（53/443 許可）

## 2. venv を使うべきか（結論）

- このテンプレート運用（フォルダごと複製・プロジェクト単位利用）では、`venv` 利用を推奨。
- 理由:
  - プロジェクトごとに依存を分離できる
  - 再現性が高い（同じ手順で同じ状態を作りやすい）
  - 依存衝突のリスクを下げられる

### 補足（Features の Python だけ使う案）

- Features のみ運用は「単発・小規模」では成立する。
- ただし複数プロジェクトを同一コンテナで扱う場合は、依存混在・再現性低下のリスクが上がる。

## 3. 「pip install 先がぶれる」の意味

- `venv` なしの場合、パッケージの入れ先が「コンテナ内の共用環境」側に寄る。
- その結果、実行時の Python とインストール先がズレると `ModuleNotFoundError` が起きやすい。
- 重要なのは容量差よりも、置き場所固定による安定運用。

## 4. 重さ（速度・容量）の違い

- 初回作成時:
  - `venv` ありは少し重くなる（`.venv` 作成処理分）
- 通常実行時:
  - 体感差はほぼ小さい
- 容量:
  - パッケージ本体はどのみち必要なので、`venv` を消しても劇的に軽くはならない
  - 主な違いは「置き場所」と「管理性」

## 5. テンプレをコピーして使うときの判断

- 方針として「フォルダごとコピーしてプロジェクト単位で使う」は相性が良い。
- ただし `.venv` は含めずにコピーし、コピー先で再構築するのが安全。

## 6. コピー前チェックリスト（3項目）

1. 機密ファイルを含めない（`.env` や認証情報を除外）
2. `.venv` を含めない（削除してからコピー）
3. `requirements.txt` を最新化して再構築可能な状態にする

## 7. 追加で気をつける点

- `devcontainer.json` の `--env-file` パスは環境依存（PC/ユーザーが変わると要修正）
- `.claude` マウント元パスも環境依存（将来の環境変更時に見直し）
- `remoteUser` を変更したら `.claude` の参照先パスも連動変更
- Claude は `/workspace` 直下ではなく、対象プロジェクト配下で起動

## 8. remoteUser のやさしい説明

- コンテナ内ユーザーごとにホームディレクトリが異なる。
  - 例: `node` -> `/home/node`
  - 例: `vscode` -> `/home/vscode`
- `remoteUser` だけ変えて設定パスを変えないと、Claude の設定ファイル参照先がズレることがある。

## 9. LF / CRLF の確認結果

- `init-firewall.sh` は LF 保存であることを確認済み。
- そのため、この点は現時点で問題なし。

## 10. 今後の運用メモ（短縮版）

- 新規プロジェクト作成時:
  - テンプレをコピー
  - `.venv` は持ち回さない
  - 必要なら `--env-file` などのパスを見直す
  - Rebuild して `.venv` を再作成
- 普段の作業:
  - 対象プロジェクト配下で Claude を起動
  - 依存追加時は `requirements.txt` を更新
